== Running a Simple Consumer Application
Now Let's see the consumer in action .

// * Go back to Devspaces IDE and clear the terminal window by typing `clear` and pressing enter.
// +
// [source,bash,role="execute",subs=attributes+]
// ----
// clear
// ----

// * Run the interactive application in the terminal.
// +
// [source,bash,role="execute",subs=attributes+]
// ----
// ./run.sh
// ----

// * It should build the application in a few seconds and display a menu of options.
// +

// [source,bash]
// ----
// ========================================
//    Kafka Java Tutorial Application
// ========================================

// Main Menu:
// 1. Run Producer
// 2. Run Consumer
// 3. Exit

// Enter your choice:
// ----

* Go back to the https://kafka-dashboard-devspaces-{user_name}.{openshift_cluster_subdomain}[interactive application, window="_application"]


* Change the auto.offset.reset value to `latest` and press enter. 
+
NOTE: This property defines what the consumer should do when there is no initial offset or offset is out of range for a consumer or consumer group* *"earliest"* = start from the beginning of the topic (read all messages from the start) *"latest"* = start from the end (only read new messages sent after consumer starts)

* Leave the rest of the values as default and hit the *Run Consumer* button to run the consumer application. This will start the consumer application and it will try to consume messages from the `tutorial-topic` topic in your Kafka cluster as configured in the `application.properties` file.
+
image::m2/consumer-interactive-app.png[]

* Observe that the consumer doesn't read any of the messages that the producer just sent. This is because we chose the `latest` value for `auto.offset.reset` and the consumer starts from the end of the topic and it will not consume any messages that are sent before the consumer starts. You can confirm the same the double checking in the streams console.

* Click on the *Stop Consumer* button to stop the consumer application.

* Navigate to the link:https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka/[streams console, window="_amqstreams"] > consumer groups and click on the *tutorial-consumer-group* 
+
image::m2/console_tutorial-consumer-group.png[]

* Observe that the overall lag for the consumer group is *zero even though the topic had some unread messages*. This is the because the consumer is reading from the end of the topic and the values of committed offset are set to the same value as end offset, which means the lag is zero.
+
image::m2/console_consumer-lag-zero.png[]


// * Hit enter and press `1` again to run the producer again. Accept all the *default values and press enter*. Make sure that the messages are sent.

// * Now hit enter and press `2` again to run the consumer again. This time let's choose the *default values* for `auto.offset.reset`, `fetch.min.bytes`, `fetch.max.wait.ms` by pressing enter for the prompts.

// * For the 'Enter number of messages' prompt. Enter the below value and press enter.
// +
// [source,bash,role="execute",subs=attributes+]
// ----
// 30    
// ----

* Click on the *Run Producer* button again to to send more messages. 

* Once the producer is done sending the messages, move to the consumer application section and change the *auto.offset.reset* value to `earliest`.
+
image::m2/consumer-auto-offset-reset.png[]

* Click on the *Run Consumer* button again to run the consumer application.

* This time, you should see all the consumer records being displayed indicating that the consumer is able to read all the messages from the topic. This is because we chose the `earliest` value for `auto.offset.reset` and the consumer starts from the beginning of the topic and it will consume all messages that are sent before the consumer starts.

* Stop the consumer application by clicking on the *Stop Consumer* button.


=== Optional exercise:Experiment with fetch.min.bytes and fetch.max.wait.ms 
You can also experiment with different values for `fetch.min.bytes` and `fetch.max.wait.ms` to see how they affect the consumer behavior.

* Run the producer again by clicking on the *Run Producer* button and send some messages to the topic.

* On the consumer application, change the `fetch.min.bytes` to the below value 
+
[source,bash,role="execute",subs=attributes+]
----
1024
----

* Change the `fetch.max.wait.ms` value to the below value.
+
[source,bash,role="execute",subs=attributes+]
----
4000
----

* Click on the *Run Consumer* button again to run the consumer application.

* Since the number of bytes sent by the producer is less than the `fetch.min.bytes` value, the consumer will wait for the `fetch.max.wait.ms` time to elapse before it fetches the messages from the topic. You should see that the consumer waits for around 4 seconds before it fetches the messages from the topic.

* Stop the consumer application by clicking on the *Stop Consumer* button.

=== Cleanup
* Make sure exit the application by pressing *ctrl+c* in the devspaces terminal window.
