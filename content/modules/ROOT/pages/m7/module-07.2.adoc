== Record Encryption in Action

Now let's see transparent record encryption in action. You will produce messages through the proxy, consume them back, and then verify that the data stored in Kafka is actually encrypted.


=== Setting Up

First, let's create a topic for customer payment data and set up a toolbox pod to interact with the proxy.

==== Create the Topic

* Create a `customer-payments` topic by applying a `KafkaTopic` resource. In the *upper terminal*, run:
+
[source,sh,role="execute",subs=attributes+]
----
cat <<'EOF' | oc apply -n kafka-{user_name} -f -
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaTopic
metadata:
  name: customer-payments
  labels:
    strimzi.io/cluster: kafka
spec:
  partitions: 1
  replicas: 3
EOF
----

* Wait a few seconds, then verify the topic was created:
+
[source,sh,role="execute",subs=attributes+]
----
oc get kafkatopic customer-payments -n kafka-{user_name}
----

You should see the topic listed with `READY: True`.

NOTE: The encryption key `KEK-customer-payments` has already been provisioned in Vault. The proxy uses the naming pattern `KEK-<topicName>`, so it will automatically use this key when encrypting records for the `customer-payments` topic.

==== Create a Toolbox Pod

* Create a Kafka toolbox container that connects through the *proxy*:
+
[source,sh,role="execute",subs=attributes+]
----
oc run -n proxy-{user_name} proxy-toolbox \
  --image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.0 \
  --restart=Never \
  --env="PROXY_BOOTSTRAP=my-cluster-cluster-ip-bootstrap.proxy-{user_name}.svc.cluster.local:9292" \
  --env="DIRECT_BOOTSTRAP=kafka-kafka-bootstrap.kafka-{user_name}.svc:9092" \
  --env="PATH=/opt/kafka/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin" \
  -- sleep infinity
----

* Wait a few seconds for the pod to start, then connect to it:
+
[source,sh,role="execute",subs=attributes+]
----
oc rsh -n proxy-{user_name} proxy-toolbox
----

Notice that we set two environment variables:

* `PROXY_BOOTSTRAP` — points to the proxy on port 9292 (encrypts/decrypts)
* `DIRECT_BOOTSTRAP` — points directly to the Kafka brokers on port 9092 (no encryption)

We will use both to demonstrate the difference.


=== Producing Encrypted Messages

Let's produce some customer payment messages *through the proxy*. The proxy will encrypt them before they reach the Kafka brokers.

* Start a console producer pointing at the *proxy*:
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-producer.sh \
  --bootstrap-server $PROXY_BOOTSTRAP \
  --topic customer-payments
----

* Type the following messages one at a time, pressing *Enter* after each:
+
[source,json]
----
{"customer":"Alice","card":"4111-XXXX-XXXX-1234","amount":99.95}
{"customer":"Bob","card":"5500-XXXX-XXXX-5678","amount":250.00}
{"customer":"Carol","card":"3400-XXXX-XXXX-9012","amount":47.50}
----

* Press *Ctrl+C* to exit the producer.

From your perspective, nothing unusual happened — you sent plaintext JSON messages to what looks like a normal Kafka cluster. But the proxy encrypted every record before storing it.


=== Consuming Through the Proxy (Decrypted)

Now let's consume those messages *through the proxy*. The proxy will decrypt them automatically.

* Start a console consumer pointing at the *proxy*:
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-consumer.sh \
  --bootstrap-server $PROXY_BOOTSTRAP \
  --topic customer-payments \
  --from-beginning
----

You should see your original plaintext messages:

[source,json]
----
{"customer":"Alice","card":"4111-XXXX-XXXX-1234","amount":99.95}
{"customer":"Bob","card":"5500-XXXX-XXXX-5678","amount":250.00}
{"customer":"Carol","card":"3400-XXXX-XXXX-9012","amount":47.50}
----

The proxy decrypted the records automatically. The consumer has no idea the data is encrypted on the broker — it just sees normal plaintext.

* Press *Ctrl+C* to exit the consumer.

=== Verifying Encryption at Rest

Now let's read the *same topic* but this time *directly from the Kafka brokers*, bypassing the proxy.

* Start a console consumer pointing at the *direct* Kafka bootstrap:
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-consumer.sh \
  --bootstrap-server $DIRECT_BOOTSTRAP \
  --topic customer-payments \
  --from-beginning
----

This time, you should see *garbled, unreadable binary data* instead of your JSON messages. The output will look something like:

[source]
----
kroxylicious.io/envelope/v2 ▒▒▒▒▒▒▒▒▒ (binary gibberish)
----

// image::m7/placeholder-consumer-direct-encrypted.png[]

*This is the proof.* The data stored on the Kafka brokers is encrypted. Only clients that connect through the proxy can read the plaintext — the proxy uses the KEK from Vault to unwrap the DEK and decrypt each record.

* Press *Ctrl+C* to exit the consumer.

==== Verify in the Streams Console

You can also see this in the {streams} console UI.

* Click https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka/[here, window="_amqstreams"] to open the {streams} console.

* Navigate to *Topics* > `customer-payments` > *Messages*.

* Notice that the message values appear as encrypted binary — the console reads directly from the brokers, so it sees the same ciphertext you saw in the terminal.

image::m7/console-encrypted-messages.png[]

Whether you use a CLI consumer or the web console, without the proxy the records are unreadable.


=== What Just Happened?

Let's recap what happened in each step:

....
                   PRODUCING (through proxy)
                   ─────────────────────────
  You (plaintext) ──► Proxy ──► Kafka Broker
                       │
                       ├─ Generated a DEK (Data Encryption Key)
                       ├─ Encrypted your message with the DEK
                       ├─ Wrapped the DEK with KEK from Vault
                       └─ Stored ciphertext + wrapped DEK in Kafka


                   CONSUMING (through proxy)
                   ─────────────────────────
  You (plaintext) ◄── Proxy ◄── Kafka Broker
                       │
                       ├─ Read ciphertext + wrapped DEK from Kafka
                       ├─ Unwrapped DEK using KEK from Vault
                       └─ Decrypted the message with the DEK


                   CONSUMING (direct — bypassing proxy)
                   ─────────────────────────────────────
  You (ciphertext!) ◄────────── Kafka Broker

                   No proxy = no decryption = unreadable data
....

To summarize:

* *Zero code changes:* producers and consumers use standard Kafka clients
* *Transparent:* connect to the proxy instead of the brokers — everything else stays the same
* *Secure key management:* KEKs never leave Vault; DEKs are per-batch and wrapped
* *Per-topic keys:* each topic can have its own KEK, enabling fine-grained control

+
[source,sh,role="execute",subs=attributes+]
----
exit
----

+
[source,sh,role="execute",subs=attributes+]
