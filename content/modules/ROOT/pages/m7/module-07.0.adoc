== Module 7: Record Encryption with Kafka Proxy
{m7}

In the previous module, you secured your Kafka cluster with authentication and authorization. But even with access controls in place, the data stored inside Kafka topics remains in *plaintext*. Anyone with access to the broker's storage — or a consumer with the right credentials — can read every message as-is.

For workloads like customer payments, personal information, or financial transactions, that is not enough. *PCI-DSS* requires that cardholder data is encrypted at rest. *GDPR* strongly recommends encryption as a technical safeguard for personal data. Either way, TLS alone isn't enough.

=== Why Not Just Use TLS?

TLS encrypts data *between* clients and brokers (in transit), but once a message lands on a broker's disk, it is stored unencrypted. Kafka does not natively support record-level encryption at rest.

You could encrypt records in your application code before producing them, but that means:

* Every producer and consumer must implement encryption/decryption logic
* Key management becomes your responsibility
* Schema Registry, Kafka Streams, and other tools see only ciphertext — breaking serialization and processing

The {streams} Proxy solves this by encrypting records *transparently* — no application code changes needed.

=== Introducing {streams} Proxy

The *{streams} Proxy* (based on the open-source https://kroxylicious.io[Kroxylicious, window="_blank"] project) is a protocol-level proxy for Apache Kafka. It sits between your Kafka clients and brokers, intercepting the Kafka protocol and applying *filters* to messages as they flow through.

Key characteristics:

* *Protocol-level:* speaks the native Kafka protocol — clients connect to the proxy exactly as they would to a Kafka broker
* *Transparent:* no SDK, no code changes, no special client libraries required
* *Filter-based:* modular architecture where filters can encrypt, transform, or validate messages
* *Operator-managed:* on OpenShift, the proxy operator manages instances using custom resources (CRDs)

For this module, we will use the *Record Encryption* filter, which encrypts message values before they reach the Kafka brokers and decrypts them when consumers read through the proxy.

=== How Envelope Encryption Works

The Record Encryption filter uses a technique called *envelope encryption*, a standard pattern used by key management systems like AWS KMS, Azure Key Vault, and HashiCorp Vault.

Here is how it works:

....
                          +-------------------+
                          |   HashiCorp Vault  |
                          |  (Transit Engine)  |
                          |                    |
                          |  KEK: "KEK-topic"  |
                          +--------+----------+
                                   |
                          wraps/unwraps DEK
                                   |
                          +--------v----------+
  Producer  ──────►       |   Kafka Proxy      |       ──────►  Kafka Broker
  (plaintext)             |                    |                (ciphertext)
                          |                    |
                          |  1. Generate DEK   |
  Consumer  ◄──────       |  2. Encrypt record |       ◄──────  Kafka Broker
  (plaintext)             |     with DEK       |                (ciphertext)
                          |  3. Wrap DEK with  |
                          |     KEK from Vault |
                          |  4. Store wrapped  |
                          |     DEK + ciphertext|
                          +-------------------+
....

Two types of keys are involved:

* *KEK (Key Encryption Key):* a master key stored in Vault's Transit engine. It never leaves Vault. One KEK is created per topic, named using the pattern `KEK-<topicName>`.
* *DEK (Data Encryption Key):* a per-batch key generated by the proxy. The DEK encrypts the actual data. The DEK itself is then encrypted (wrapped) using the KEK and stored alongside the ciphertext in Kafka.

When a consumer reads through the proxy, the process reverses: the proxy retrieves the wrapped DEK from the record, asks Vault to unwrap it using the KEK, and then decrypts the record — all transparently.

=== Beyond Record Encryption

Record encryption is one use case, but a protocol-level proxy can do much more. Because it sits between clients and brokers, it can apply logic to the Kafka protocol without touching application code. Some examples:

* *Multi-tenancy:* present isolated virtual clusters backed by a single shared Kafka cluster
* *Schema enforcement:* reject messages that don't match an expected schema before they reach the broker
* *Observability:* expose protocol-level metrics and traces without instrumenting individual applications

In this module, we focus on record encryption — but the proxy's filter model means it can be used for much more than just encryption.
