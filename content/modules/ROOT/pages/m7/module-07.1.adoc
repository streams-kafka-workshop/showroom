== Exploring the Encryption Setup

Before we start encrypting records, let's look at the components that have been pre-configured on your OpenShift cluster and how they connect.

=== The Kafka Cluster

Your Kafka cluster runs in the `kafka-{user_name}` namespace with 3 brokers managed by {streams}.

* Click https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka/[here, window="_amqstreams"] to open the {streams} console.

* Login anonymously to the console and note the cluster overview — 3 brokers, KRaft mode (no ZooKeeper).

// image::m7/placeholder-streams-console-overview.png[]

This is the same Kafka cluster you have been using throughout the workshop. The key difference in this module is *how* clients connect to it — through the {streams} Proxy instead of directly.


=== The {streams} Proxy

The proxy runs in the `proxy-{user_name}` namespace, managed by the {streams} Proxy operator. It is configured using a set of custom resources (CRDs) that together define how the proxy connects to Kafka and what filters it applies.

Let's examine each one.

* In the *upper terminal*, log in to the OpenShift cluster:
+
[source,sh,role="execute",subs=attributes+]
----
{login_command}
----

* Type `y` and press enter when prompted `Use insecure connections? (y/n):`

==== KafkaProxy

The `KafkaProxy` resource creates the proxy instance itself.

* Run the following command to view it:
+
[source,sh,role="execute",subs=attributes+]
----
oc get kafkaproxy simple -n proxy-{user_name} -o yaml
----

This is the simplest resource — it tells the operator to create a proxy pod. The actual configuration is in the other resources below.

==== KafkaService

The `KafkaService` tells the proxy where the *real* Kafka cluster is.

* View the KafkaService:
+
[source,sh,role="execute",subs=attributes+]
----
oc get kafkaservice my-cluster -n proxy-{user_name} -o yaml
----

Notice the `bootstrapServers` field pointing to `kafka-kafka-bootstrap.kafka-{user_name}.svc:9092` — this is the direct address of your Kafka cluster that the proxy will forward traffic to.

==== KafkaProtocolFilter

This is where the encryption is configured. The `KafkaProtocolFilter` defines the Record Encryption filter.

* View the encryption filter:
+
[source,sh,role="execute",subs=attributes+]
----
oc get kafkaprotocolfilter encryption -n proxy-{user_name} -o yaml
----

Key fields to note:

* `type: io.kroxylicious.filter.encryption.RecordEncryption` — the filter that encrypts and decrypts records
* `kms: VaultKmsService` — uses HashiCorp Vault as the key management service
* `vaultTransitEngineUrl` — the Vault Transit API endpoint where encryption keys are managed
* `selector: TemplateKekSelector` with template `KEK-$(topicName)` — this means each topic gets its own encryption key, named `KEK-<topicName>`. For example, a topic named `customer-payments` uses the key `KEK-customer-payments`

==== VirtualKafkaCluster

The `VirtualKafkaCluster` ties everything together — the proxy, the target Kafka cluster, and the encryption filter.

* View the VirtualKafkaCluster:
+
[source,sh,role="execute",subs=attributes+]
----
oc get virtualkafkacluster my-cluster -n proxy-{user_name} -o yaml
----

Notice how it references:

* `proxyRef: simple` — the KafkaProxy instance
* `targetKafkaServiceRef: my-cluster` — the real Kafka cluster
* `filterRefs: encryption` — the encryption filter to apply
* `ingresses` — how clients connect to this virtual cluster

==== KafkaProxyIngress

The ingress exposes the proxy as a Kubernetes service so clients can connect to it.

* View the ingress:
+
[source,sh,role="execute",subs=attributes+]
----
oc get kafkaproxyingress cluster-ip -n proxy-{user_name} -o yaml
----

This creates a ClusterIP service at `my-cluster-cluster-ip-bootstrap.proxy-{user_name}.svc.cluster.local` on port `9292`. Clients connect to this address instead of the Kafka brokers directly.

* You can verify the service exists:
+
[source,sh,role="execute",subs=attributes+]
----
oc get svc -n proxy-{user_name}
----

=== HashiCorp Vault

HashiCorp Vault runs in the `kms` namespace and provides the *Transit secrets engine* — a cryptographic service that manages encryption keys without exposing the key material.

The proxy uses Vault to:

* Store KEKs (Key Encryption Keys) — one per topic
* Wrap and unwrap DEKs (Data Encryption Keys) used for actual record encryption

For this module, the key `KEK-customer-payments` has already been provisioned for you.

NOTE: The proxy handles all communication with Vault transparently. You do not need to interact with Vault directly.


=== Putting It All Together

Here is how the resources connect:

....
KafkaProxy ("simple")
    │
    ├── KafkaProxyIngress ("cluster-ip")
    │       → exposes proxy on port 9292
    │
    └── VirtualKafkaCluster ("my-cluster")
            │
            ├── targetKafkaServiceRef → KafkaService ("my-cluster")
            │       → forwards to kafka-kafka-bootstrap:9092
            │
            └── filterRefs → KafkaProtocolFilter ("encryption")
                    → RecordEncryption using Vault KEKs
....

When a client connects to the proxy on port 9292, the `VirtualKafkaCluster` routes the traffic to the real Kafka cluster on port 9092, applying the encryption filter.
